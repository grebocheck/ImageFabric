# HFabric — Project Audit (2026-06-11)

Scope: full source review of `backend/app` (~8.1k lines), `frontend/src` (~8.5k lines),
tests, CI, scripts, repo hygiene, and runtime configuration. Both test suites were
executed during the audit: **backend 107/107 passed**, **frontend 36/36 passed**.

## Verdict

**Overall: B (7.5/10).** A genuinely well-engineered single-user local app with an
unusually disciplined memory architecture and a real (if young) test/CI safety net.
The biggest risks are not in the model code at all: an **unauthenticated API that the
local `.env` currently binds to `0.0.0.0:80`**, a hand-rolled DB migration path, two
~1k-line monolith modules, and an entire phase (voice) that is wired but never
validated end-to-end.

| Area | Grade | Summary |
|------|-------|---------|
| Architecture | A− | Arbiter/sysmon invariants, single worker, event bus, registry — clear seams, enforced in one place |
| Memory discipline | A | Budget guard, learned profiles, warm-eviction, leak runners; the core product promise is actually engineered |
| Tests & CI | B | Good foundation (143 tests, stub-mode integration, CI on push/PR) but most API routers and all GPU paths are untested |
| Security | D | No auth on an API that deletes files, spawns processes, and reaches the desktop; currently exposed on `0.0.0.0:80` |
| Code health | B− | Helper extraction started, but ChatPanel (1070), image_diffusers (1435), VoicePanel (749) remain monoliths; no frontend lint |
| Data layer | C+ | SQLite + ad-hoc `ALTER TABLE` migration; no backup story; no structured logs for post-mortems |
| Docs | A− | README + ROADMAP + hard-won-facts are excellent and current; knob table is hand-maintained |
| Distribution | C | Dev-mode only (uvicorn + Vite dev server); no production build serving, no pinned GPU lockfile |

## Strengths (keep doing this)

1. **The memory invariants are real code, not aspiration.** `GpuArbiter._guard_budget`
   refuses loads pre-emptively, evicts warm models one-by-one before refusing, and
   publishes structured reasons (`arbiter.py:198`). Learned per-model profiles replace
   static heuristics. This is the hardest part of the product and it is the best part
   of the codebase.
2. **STUB mode as a first-class design** makes the entire queue→swap→gallery pipeline
   hermetically testable on CI without a GPU (`test_stub_integration.py`).
3. **Single-worker serialization** makes GPU correctness structural rather than
   lock-sprinkled; orphaned RUNNING jobs are requeued on restart (`scheduler.py:160`).
4. **Shared pure logic** (`select_in_tier`/`plan_queue`) keeps the swap-plan preview
   and the live scheduler provably in sync.
5. **Repo hygiene**: models/binaries/data properly untracked; `.gitattributes`,
   `.editorconfig`, ruff config committed; honest commit messages on recent work.
6. **Upload path-traversal guard** centralized in one place (`util/uploads.py`) with
   an opaque-token-only rule.

## Weaknesses (ranked by risk)

### W1 — Unauthenticated API exposed beyond localhost (critical)
The local `.env` sets `HFAB_HOST=0.0.0.0` and `HFAB_PORT=80`. There is **no
authentication anywhere**. Anyone on the LAN can: delete gallery images
(`DELETE /api/images/{id}`), bulk-export files, launch/stop the w-okada subprocess
(`api/voice.py`), relaunch llama-server with arbitrary ctx, run LLM/image jobs, and —
notably — `POST /api/images/{id}/reveal` spawns `explorer /select,<path>` on the
desktop (`gallery.py:188`). CORS does not protect against non-browser clients; it only
restricts browsers. Either bind to `127.0.0.1` (the code default — the `.env` override
is the problem) or add a token check before any LAN exposure.

### W2 — Voice phase (P6) wired but unvalidated (high)
All four P6 sub-items are implemented but none has been validated against a real audio
stream (per ROADMAP). 528 lines of `api/voice.py` including subprocess lifecycle and a
GPU "voice lane" that parks the queue — this is the largest body of unverified code in
the project, and it holds module-level process state (`_proc`).

### W3 — Hand-rolled schema migration (medium, compounding)
`db/session.py:_ensure_image_columns` does `PRAGMA table_info` + `ALTER TABLE` for
three columns. Every future column grows this function; there is no version stamp, no
downgrade, no second table covered. Fine at 5 tables, brittle at 10. Alembic (or a tiny
versioned-migration table) is cheap now and expensive later.

### W4 — Test coverage is wide at the core, absent at the edges (medium)
Covered well: scheduler, sysmon, arbiter budget, gallery service, img2img params, LLM
config, tool parsing. **Not covered at all**: `api/chat.py` (312 lines), `api/rag.py`,
`api/tts.py`, `api/transcription.py`, `api/vision.py`, `api/voice.py`,
`chat_service.py`, `embedding_service.py`, notes/presets/code routers. No coverage
measurement in CI, so regressions in the untested half are invisible. Frontend has 36
tests but none for ChatPanel/Gallery/QueuePanel flows.

### W5 — Monolith modules (medium)
`image_diffusers.py` (1435 lines, all families + LoRA + recycling + img2img + inpaint
in one file), `ChatPanel.tsx` (1070), `VoicePanel.tsx` (749), `ImageComposer.tsx`
(697), `Gallery.tsx` (632). P11.1 extraction started well (helpers + tests) but the
hard splits remain. Every new image family makes the diffusers file worse.

### W6 — Reproducibility of the GPU environment (medium)
`requirements-gpu.txt` mixes pins with install *instructions in comments* (torch cu128
index, nunchaku wheel URL). A disk failure or new machine means re-deriving the
environment from comments and memory. No lockfile/constraints file for the verified
M0 stack.

### W7 — No observability for post-mortems (low-medium)
Events flow over the bus to the UI, but nothing is persisted: no rotating file log, no
record of arbiter refusals/swaps/errors after the process exits. Debugging "it hung
overnight" relies on a live console.

### W8 — Dev-only run story (low)
`run.bat`/`run.sh` start uvicorn + Vite dev server. There is no production mode
(built frontend served by FastAPI on one port), which also complicates the W1 fix
(two ports to protect instead of one).

### W9 — Small hygiene items (low)
- `frontend/tsconfig.tsbuildinfo` (build artifact) is committed.
- No frontend eslint/prettier (P11.2 tail), so style drift is unchecked in CI.
- `types.ts` (515 lines) mirrors backend schemas by hand — drift risk; OpenAPI codegen
  would remove the class of bugs.
- Settings drawer is read-only; knob table in README is hand-maintained (P11.3 tail).
- Job errors surface raw `repr(exc)` to the UI — acceptable locally, but worth a
  friendly-message layer once errors are logged (W7).
